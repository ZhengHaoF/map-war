# 外交消息总线设计方案（三段式）

> 对应架构决策 §3.3.4 / Q6。状态：⬜ 未实现 → 本文档为落地设计稿。
> 设计原则：**叠加在 Kernel 之上，非取代**；外交对话本身不改世界态，只有世界 AI 收口产出的结构化事件经 `applyEvent` 才改。

---

## 1. 定位与边界

### 1.1 外交消息总线是什么

一条**独立的对话编排路径**，处理玩家发起的跨阵营交互（使团、谈判、宣战、结盟、停战等）。与常规 P1-P4 回合循环**并行存在**：

- **P1-P4 循环**：玩家下军事指令 → 结束回合 → 各政权 AI 决策 → 世界 AI 结算。
- **外交总线**：玩家发起外交请求 → 三段式对话 → 世界 AI 收口产出事件 → 事件入队执行。

外交总线**可在 P1 阶段随时触发**（玩家不结束回合也能发起外交），也可在 P4 结算后由世界 AI 主动触发（如某势力向玩家宣战）。

### 1.2 与电报系统的关系

| 维度 | 电报（Telegram） | 外交消息总线 |
|------|-----------------|-------------|
| 目的 | 氛围/角色扮演/信息传递 | **改变世界态**（结盟/宣战/停战/贸易） |
| 产出 | 文本消息（不进 eventLog） | 叙事 + 结构化事件（进 eventLog，走 applyEvent） |
| 触发 | AI 搭车 P3/P4 输出（零额外 LLM） | 玩家主动发起（每次至少 2 次 LLM：路由 + 收口） |
| 多轮 | 玩家回信 = 1 次 LLM | 目标阵营 AI 可多轮协商 |
| 关系 | 电报可作为外交总线的**前置**（先发电报试探，再正式派使团） | 外交结果可触发电报（宣战通电全国） |

### 1.3 不做什么（v1 边界）

- 不做 AI 势力之间的自主外交（仅玩家发起）
- 不做实时谈判 UI（用 toast + 电报面板呈现结果）
- 不做贸易/经济协定（仅军事外交：宣战/停战/结盟/解除结盟/请求通行）
- 不做多边谈判（一次只对一个目标阵营）

## 2. 三段式流程详细设计

### 2.1 第一段：世界 AI 校验 + 路由

**触发**：玩家在 PlayerAiPanel 输入自然语言，AI 分类器判定为「外交意图」。

**输入**：
```typescript
interface DiplomacyRouteInput {
  playerInput: string        // 玩家原话："派使团去太原，跟阎锡山谈联合抗日"
  playerFaction: Owner       // SCC
  currentDate: string        // 1931-04-01
  worldOverview: string      // buildWorldOverview() 的紧凑城市全景
  recentDiplomacy: string    // 近期外交事件摘要
}
```

**LLM 调用**：`invokeDiplomacyRoute(input)` → 1 次 callLlm

**输出**：
```typescript
interface DiplomacyRouteResult {
  valid: boolean             // 请求是否合法（目标存在？玩家有权？）
  rejectReason?: string      // 不合法时的拒绝理由
  targetFaction: Owner       // 路由目标（SHX）
  targetLeader: string       // 领袖名（阎锡山）
  intent: DiplomacyIntent    // 结构化意图
  narrative: string          // 世界 AI 的即时叙事（"使团已启程前往太原…"）
}

type DiplomacyIntent =
  | 'alliance'       // 结盟
  | 'peace'          // 停战/和谈
  | 'war'            // 宣战
  | 'breakAlliance'  // 解除结盟
  | 'passage'        // 请求借道/通行
  | 'trade'          // 贸易（v2）
  | 'custom'         // 其他自定义请求
```

**校验规则（代码级，零 LLM）**：
- 目标阵营必须存活（`activeFactions` 包含）
- 玩家不能对自己发起外交
- 宣战/结盟等意图需目标阵营存在

**路由失败处理**：`valid=false` → 推 toast 告知玩家 → 结束（不进入第二段）。

### 2.2 第二段：目标阵营 AI 回应（可多轮）

**输入**：
```typescript
interface DiplomacyReplyInput {
  faction: Owner             // SHX（目标阵营）
  intent: DiplomacyIntent    // alliance
  playerMessage: string      // 玩家原话
  routeNarrative: string     // 第一段世界 AI 的叙事
  factionContext: string     // buildFactionContext(SHX)：该势力视角的城市/兵力/邻接
  negotiationHistory: string // 本轮对话历史（多轮时累积）
  currentRelation: DiplomacyLevel  // 当前外交关系
  round: number              // 第几轮（从 1 开始）
}
```

**LLM 调用**：`invokeDiplomacyReply(input)` → 每轮 1 次 callLlm

**输出**：
```typescript
interface DiplomacyReplyResult {
  stance: 'accept' | 'reject' | 'counter' | 'stall'
  reply: string              // 阵营领袖的回复文本（半文言，性格化）
  counterOffer?: string      // stance='counter' 时的反提议
  conditions?: string[]      // 接受/反提议的条件列表
  done: boolean              // 是否结束协商（true = 进入第三段）
}
```

**多轮规则**：
- `done=true`：协商结束，进入第三段收口
- `done=false` + `stance='counter'`：玩家可继续回应（回到 PlayerAiPanel 输入）
- `done=false` + `stance='stall'`：拖延（"容我考虑三日"），世界 AI 可插入时间推进
- **轮次上限**：不设硬上限（§6.2 拍板），但 UI 提示"已协商 N 轮"

**玩家多轮参与方式**：
- 第二段 `done=false` 时，将目标阵营回复推入电报面板（direct 频道）+ toast
- 玩家在 PlayerAiPanel 继续输入 → 再次进入第二段（round++）
- 玩家也可选择"不再协商" → 强制 done=true，以当前状态进入第三段

### 2.3 第三段：世界 AI 收口（双件套）

**输入**：
```typescript
interface DiplomacySettleInput {
  session: DiplomacySession  // 完整协商记录
  playerFaction: Owner
  targetFaction: Owner
  intent: DiplomacyIntent
  finalStance: 'accept' | 'reject' | 'counter'  // 最终结果
  worldOverview: string
}
```

**LLM 调用**：`invokeDiplomacySettle(input)` → 1 次 callLlm

**输出（双件套，缺一不可）**：
```typescript
interface DiplomacySettleResult {
  narrative: string          // 2-4 句叙事文案（"阎锡山与川军歃血为盟…"）
  events: DiplomacyEvent[]   // 结构化事件（走 applyEvent）
  telegrams?: {              // 可选：触发的电报（宣战通电等）
    channel: 'direct' | 'world'
    from: string
    content: string
  }[]
}
```

**关键约束**：
- `narrative` + `events` 必须同时产出（§3.3.4 拍板："否则外交不改世界态，回到 AI 只会放烟花"）
- `events` 中的每条事件都走 `applyEvent`（Kernel 唯一写者）
- 拒绝也算 done：`events` 可为空数组（无世界态变更），但 `narrative` 必须有

## 3. 数据模型扩展

### 3.1 外交关系矩阵（store 新增）

```typescript
// stores/game.ts 新增
type DiplomacyLevel = 'ALLIED' | 'FRIENDLY' | 'NEUTRAL' | 'HOSTILE' | 'WAR'

// 关系矩阵：relations[from][to] = DiplomacyLevel（对称）
const relations = ref<Record<string, Partial<Record<string, DiplomacyLevel>>>>({})

// 初始值由 initWorld() 灌入（从 owners.ts 种子数据）
// 查询接口
function getRelation(a: Owner, b: Owner): DiplomacyLevel {
  return relations.value[a]?.[b] ?? 'NEUTRAL'
}
```

**initWorld() 重置**：`relations.value = {}` → 从种子数据灌入初始关系。

**replay 安全**：外交关系变更事件（declareWar/makePeace/formAlliance/breakAlliance/diplomacyRelation）经 `applyEvent` 修改 `relations`，replay 时重放即可恢复。

### 3.2 新增 GameEvent 类型

```typescript
// 外交过程事件（记录协商过程，replay 时可还原对话）
| { type: 'diplomacyStart'; sessionId: string; actor: Owner; target: Owner; intent: string }
| { type: 'diplomacyReply'; sessionId: string; actor: Owner; stance: string; reply: string }
| { type: 'diplomacySettle'; sessionId: string; result: string }

// 外交结果事件（改世界态）
| { type: 'declareWar'; actor: Owner; target: Owner }
| { type: 'makePeace'; actor: Owner; target: Owner }
| { type: 'formAlliance'; actor: Owner; target: Owner }
| { type: 'breakAlliance'; actor: Owner; target: Owner }
| { type: 'diplomacyRelation'; from: Owner; to: Owner; relation: DiplomacyLevel }
```

**applyEvent 处理**：
- `diplomacyStart/Reply/Settle`：仅记录到 eventLog，不改世界态（类似 narrative）
- `declareWar`：`setRelation(actor, target, 'WAR')` + 触发宣战动画
- `makePeace`：`setRelation(actor, target, 'NEUTRAL')`
- `formAlliance`：`setRelation(actor, target, 'ALLIED')`
- `breakAlliance`：`setRelation(actor, target, 'NEUTRAL')`
- `diplomacyRelation`：通用关系设置（世界 AI 批量事件用）

### 3.3 DiplomacySession（协商会话）

```typescript
// composables/useDiplomacyBus.ts 模块级
interface DiplomacySession {
  id: string                 // 'diplo_' + timestamp
  status: 'initiated' | 'routing' | 'negotiating' | 'settling' | 'done' | 'failed'
  playerFaction: Owner
  targetFaction: Owner
  intent: DiplomacyIntent
  playerInput: string
  rounds: DiplomacyRound[]   // 多轮对话记录
  routeResult?: DiplomacyRouteResult
  settleResult?: DiplomacySettleResult
  createdAt: string          // 游戏日期
}

interface DiplomacyRound {
  round: number
  playerMessage?: string     // 玩家本轮发言（第一轮为原始输入）
  factionReply: DiplomacyReplyResult
  timestamp: string          // 游戏日期
}
```

**生命周期**：模块级 ref，不进 store（非世界态）；协商完成后产出事件进 eventLog。存档时不需要持久化进行中的 session（中断即失败，玩家重新发起）。

## 4. 代码落点与接口设计

### 4.1 新文件：`composables/useDiplomacyBus.ts`

外交消息总线的核心编排器（模块级单例，同 useAgentKernel 模式）：

```typescript
// 模块级状态
const session = ref<DiplomacySession | null>(null)
const loading = ref(false)
const phase = ref<'idle' | 'routing' | 'negotiating' | 'settling' | 'done' | 'failed'>('idle')

// 公开 API
export function useDiplomacyBus() {
  return {
    session,
    loading,
    phase,
    /** 玩家发起外交请求（第一段入口） */
    startDiplomacy(playerInput: string): Promise<DiplomacyRouteResult>,
    /** 玩家继续协商（第二段多轮） */
    continueNegotiation(playerReply: string): Promise<DiplomacyReplyResult>,
    /** 玩家强制结束协商（跳到第三段） */
    forceSettle(): Promise<DiplomacySettleResult>,
    /** 取消当前协商 */
    cancelDiplomacy(): void,
  }
}
```

### 4.4 触发入口：PlayerAiPanel 分流

```typescript
// useAiOrchestrator.ts user 模式改造
async function handleUserInput(text: string) {
  // 1. 先做意图分类（轻量，可用关键词 + 现有 AI）
  const intent = classifyPlayerIntent(text)  // 'military' | 'diplomacy' | 'query'

  if (intent === 'diplomacy') {
    // 走外交总线
    const bus = useDiplomacyBus()
    await bus.startDiplomacy(text)
  } else {
    // 走现有军事指令路径
    // ...existing logic...
  }
}
```

**意图分类方案**（零额外 LLM，代码级）：
```typescript
const DIPLOMACY_KEYWORDS = [
  '使团', '外交', '谈判', '结盟', '联盟', '停战', '和谈', '宣战',
  '借道', '通行', '协议', '条约', '派使者', '联络', '交涉',
  '联合', '合作', '求和', '投降', '劝降',
]
function classifyPlayerIntent(text: string): 'military' | 'diplomacy' | 'query' {
  if (DIPLOMACY_KEYWORDS.some(k => text.includes(k))) return 'diplomacy'
  return 'military'  // 默认走军事（现有路径）
}
```

> 备选：让世界 AI 在第一段同时做意图分类（已有 LLM 调用，不额外花钱）。但关键词分流更快、更确定，推荐先用关键词，后续可升级为 LLM 分类。

## 5. 与现有系统的集成点

### 5.1 与 P1-P4 回合循环的关系

- 外交总线**在 P1 阶段触发**（玩家输入外交意图时）
- 外交产出的事件**立即入队执行**（不等结束回合）
- P4 结算时，世界 AI 能看到本轮的外交事件（已在 eventLog 中）
- 外交不阻塞 P1-P4：玩家发完外交请求后仍可继续下军事指令

### 5.2 与电报系统的联动

- 外交第二段的目标阵营回复 → 同时推入电报面板（direct 频道），玩家可在电报 UI 中看到"对话记录"
- 外交收口若产出宣战 → 自动推 world 频道通电（"XX 对 XX 宣战！"）
- 电报面板中可显示外交状态标记（如"协商中…"）

### 5.3 与调度器的关系

- 外交收口产出的 `events[]` → 转换为 `GameOrder[]` → `scheduler.submit()` → `advance()` 演出
- 宣战/结盟等事件触发对应动画（复用现有 `executeOrder` 的 `battle`/`capture` 动画，或新增 `diplomacy` 动画）

### 5.4 与世界 AI 批量事件的关系

- 世界 AI 在 P3 批量生成 minor 势力事件时，可产出 `diplomacyRelation` 事件（如"马家军与晋系关系恶化"）
- 这走现有的 `invokeWorldAIBatch` → `validateFactionOrder` 路径，只需扩展契约支持 `diplomacyRelation` 指令

## 6. 提示词设计要点

### 6.1 第一段（世界 AI 校验路由）

```
你是民国军阀推演游戏的「世界 AI」外交官。玩家扮演{playerFactionLabel}。
玩家发来外交请求，你需要：
1. 判断请求是否合法（目标势力是否存在、玩家是否有权发起）
2. 识别外交意图（结盟/停战/宣战/借道/其他）
3. 确定目标势力
4. 给出一句即时叙事（如"使团已启程…"）

当前局势：{worldOverview}
近期外交：{recentDiplomacy}
玩家原话：「{playerInput}」

返回 JSON：
{ "valid": true, "targetFaction": "SHX", "targetLeader": "阎锡山", "intent": "alliance", "narrative": "…" }
若不合法：{ "valid": false, "rejectReason": "…" }
```

### 6.2 第二段（目标阵营 AI）

```
你是「{factionLabel}」的领袖{leaderName}，性格{personality}。
当前局势：{factionContext}
与{playerFactionLabel}的当前关系：{currentRelation}

{playerFactionLabel}向你提出外交请求：{intent 描述}
使者转达：「{playerMessage}」
{negotiationHistory ? '此前协商记录：\n' + negotiationHistory : ''}

以你的性格和立场回应（80-150字，半文言）。决定：
- accept：同意（可附条件）
- reject：拒绝（给理由）
- counter：反提议（给出你的条件）
- stall：拖延（需要时间考虑）

返回 JSON：
{ "stance": "counter", "reply": "联合抗日？哼…", "counterOffer": "需你川军先让出汉中", "conditions": ["让出汉中"], "done": false }
```

### 6.3 第三段（世界 AI 收口）

```
你是民国军阀推演游戏的「世界 AI」叙事者。一场外交协商已结束。

协商双方：{playerFactionLabel} vs {targetFactionLabel}
外交意图：{intent}
最终结果：{finalStance}
协商过程：{sessionSummary}
当前局势：{worldOverview}

产出双件套：
1. narrative：2-4 句叙事（半文言，有历史感）
2. events：结构化事件数组（改世界态）

事件类型：
- declareWar: { "type": "declareWar", "actor": "SHX", "target": "SCC" }
- makePeace: { "type": "makePeace", "actor": "SCC", "target": "SHX" }
- formAlliance: { "type": "formAlliance", "actor": "SCC", "target": "SHX" }
- breakAlliance: { "type": "breakAlliance", "actor": "SCC", "target": "SHX" }
- diplomacyRelation: { "type": "diplomacyRelation", "from": "SCC", "to": "SHX", "relation": "FRIENDLY" }

若拒绝：events 为空数组 []，但 narrative 必须有。

返回 JSON：
{ "narrative": "…", "events": [...], "telegrams": [...] }
```

## 8. 成本估算

| 场景 | LLM 调用次数 |
|------|-------------|
| 外交请求被拒（第一段即失败） | 1 次 |
| 单轮协商成功 | 3 次（路由 + 回应 + 收口） |
| 三轮协商后成功 | 5 次（路由 + 3×回应 + 收口） |
| 玩家强制结束 | 3 次（路由 + N×回应 + 收口） |

对比常规回合（P3 并行 N 次 + P4 一次），外交总线的额外成本可控（每次外交 3-5 次 LLM）。

## 9. 风险与对策

| 风险 | 对策 |
|------|------|
| 世界 AI 收口不产出 events（"只放烟花"） | 代码级兜底：若 events 为空且 finalStance='accept'，自动补一条 `diplomacyRelation` |
| 目标阵营 AI 永远拒绝（死循环） | 不设轮次上限但 UI 提示；玩家可 forceSettle 强制收口 |
| 外交事件 replay 时重复改关系 | applyEvent 幂等：setRelation 是覆盖式赋值，重放安全 |
| 外交与军事冲突（刚结盟就开打） | 世界 AI 校验层拦截：结盟状态下不允许 attack 指令（validateFactionOrders 扩展） |
| 玩家用外交绕过军事限制 | 第一段校验：外交不能直接产出军事指令（events 仅限外交类型） |
