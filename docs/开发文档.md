# 开发规范

## 项目概述

本项目是一个中国战争 AI 游戏，使用 Vue 3 + Pixi.js 实现地图交互。

## UI 组件库

### 设计规范

> ⚠️ **视觉风格以羊皮纸水墨风为准**（与 `App.vue` 的 `--paper`/`--ink`/`--cinnabar` 主题变量、地图界面、`择势` 弹窗一致）。早期版本曾用深色玻璃拟态（`rgba(20,20,40)` 等），现已废弃，下方为当前规范。

| 属性 | 值（CSS 变量） | 说明 |
|------|------|------|
| 背景色 | `var(--paper-panel)` | 羊皮纸面板底色 |
| 次级背景 | `var(--paper-faint)` / `var(--paper-darker)` | 卡片、输入框底 |
| 边框色 | `rgba(138, 109, 75, 0.4)` | 褐边（与 `--brown-line` 同系） |
| 文字主色 | `var(--ink)` / `var(--ink-strong)` | 墨色 |
| 文字次色 | `var(--ink-soft)` | 浅墨 |
| 文字辅助色 | `var(--ink-muted)` | 淡墨 |
| 主题强调 | `var(--cinnabar)` / `var(--cinnabar-ink)` | 朱砂红（高亮、激活、聚焦） |
| 悬停高亮 | `var(--paper-hi)` 背景 + `var(--cinnabar-ring)` 内描边 | 聚焦态用朱砂环 |
| 危险操作色 | `var(--danger-bg)` 背景 + `var(--cinnabar)` 边框 | 危险按钮 |
| 字体 | `var(--font-kai)`（楷）/ `var(--font-xing)`（行楷，标题） | 中文衬线 |
| 圆角 | `var(--radius-xs)` 2px / `var(--radius-sm)` 4px / `var(--radius-md)` 6px / `var(--radius-lg)` 12px / `var(--radius-pill)` 999px | 方角四档（微件→大容器）+ 圆形一档（toggle） |
| 阴影 | `0 6px 22px rgba(60, 40, 15, 0.28)` + 内描边 | 替代毛玻璃，呈纸片浮起感 |

> 另有派生变量（输入框 `--paper-input`、卡片高亮 `--paper-hi2`、深色阶梯 `--paper-dark` / `--paper-darkest`、危险系 `--danger-bg2` / `--danger-ink` / `--danger-text` 等）定义于 `App.vue` 的 `:root`，按需取用；本表只列最常用者。

### GameButton

通用游戏按钮，支持两种视觉变体：

- 默认变体（玻璃）：深色半透明玻璃按钮（`background: rgba(0,0,0,0.6)` + `backdrop-filter`），**仅用于浮在地图之上的悬浮按钮**（图层切换、调试等），借深色与地图形成对比、保证可读；
- `parchment` 变体：羊皮纸/古籍风（与全局纸张主题一致），用于面板内、HUD 上的常规操作。

> 全局视觉规范以羊皮纸水墨风为准（见上方「设计规范」）。玻璃变体是悬浮于地图的特例，不是面板内的默认风格——面板内请统一用 `parchment`。

内置 `active`（**朱砂红**高亮，非蓝色）、`danger`（红色悬停）两种语义态，以及 `tooltip` 悬停提示。图标与文字并排，图标经 `:deep(svg)` 防止被压缩。

```vue
<script setup lang="ts">
import IconSword from '~icons/tabler/sword'
</script>

<template>
<GameButton :active="isActive" :danger="isDanger" :parchment="isPaper" :tooltip="'说明'" @click="handler">
  <component :is="IconSword" /> 按钮文本
</GameButton>
</template>
```

| Prop | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| active | Boolean | false | 激活状态（朱砂红 `rgba(178,58,46,.85)` 高亮） |
| danger | Boolean | false | 危险状态（红色悬停，不默认红底） |
| parchment | Boolean | false | 羊皮纸变体（浅底墨字，替代默认深色玻璃） |
| tooltip | String | `''` | 悬停提示文字（右侧浮出，羊皮纸气泡） |

### GameModal

弹窗/浮动面板组件，自动 Teleport 到 body。支持普通居中弹窗和可拖动的浮动面板两种模式。

```vue
<!-- 普通弹窗（居中、带遮罩） -->
<GameModal :visible="showModal" title="标题" @close="showModal = false">
  <p>内容</p>
</GameModal>

<!-- 可拖动的浮动面板（无遮罩，自由定位） -->
<GameModal
  :visible="showPanel"
  title="面板"
  :draggable="true"
  :overlay="false"
  :init-x="160"
  :init-y="160"
  @close="showPanel = false"
>
  <p>可拖动的面板内容</p>
</GameModal>
```

| Prop | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| visible | Boolean | false | 是否显示 |
| title | String | `''` | 标题文本 |
| minWidth | String | `'280px'` | 最小宽度 |
| draggable | Boolean | false | 是否可拖动 |
| overlay | Boolean | true | 是否显示遮罩层 |
| initX | Number | 160 | 初始 X 位置（`draggable` 时生效） |
| initY | Number | 160 | 初始 Y 位置 |

| Event | 说明 |
|-------|------|
| close | 关闭弹窗 |

**拖动说明：** 按住标题栏拖拽，标题栏在 `draggable` 时显示 `grab/grabbing` 光标。`overlay=false` 时遮罩透明且不阻塞地图交互。

## 图标库（Tabler Icons）

本项目使用 **unplugin-icons + @iconify-json/tabler** 作为图标方案，编译期按需导入、零运行时体积。

### 安装

已在 devDependencies 中安装，无需额外操作：

```json
"devDependencies": {
  "unplugin-icons": "^23.0.1",
  "@iconify-json/tabler": "^1.2.35"
}
```

### 静态用法（推荐：图标名固定）

用 `~icons/tabler/{图标名}` 显式导入组件，然后通过 `<component :is>` 渲染：

```vue
<script setup lang="ts">
import IconSword from '~icons/tabler/sword'

// 也可通过查表动态使用
import IconFlag from '~icons/tabler/flag'
import IconX from '~icons/tabler/x'
import type { Component } from 'vue'

const ICONS: Record<string, Component> = {
  flag: IconFlag,
  x: IconX,
}
</script>

<template>
  <!-- 直接引用 -->
  <component :is="IconSword" :size="16" />

  <!-- 通过查表动态切换 -->
  <component :is="ICONS['flag']" :size="16" />
</template>
```

> ⚠️ **不要使用 `<i-tabler-xxx>` 标签写法。** `unplugin-icons` 的 `compiler: 'vue3'` 选项**不会在运行时自动注册全局组件**，Vue 会将其当作未知自定义元素处理，导致图标不渲染。必须用 `~icons/tabler/xxx` 显式 import 后通过 `<component :is>` 渲染。

### 动态用法（图标名来自数据/配置）

右键菜单等需要根据数据字段动态渲染图标时，先静态导入所有用到的图标，再通过查表映射：

```vue
<script setup lang="ts">
import type { Component } from 'vue'
import IconInfoCircle from '~icons/tabler/info-circle'
import IconSearch from '~icons/tabler/search'

const ICONS: Record<string, Component> = {
  'info-circle': IconInfoCircle,
  search: IconSearch,
}
</script>

<template>
  <component :is="ICONS[item.icon]" :size="16" />
</template>
```

### 世界国家数据

定义在 `src/data/worldCountries.ts`：

```ts
interface CountryData {
  /** 唯一ID，与 iso_a3 相同 */
  id: string
  /** ISO 国家代码（与世界.geojson 对应） */
  iso_a3: string
  /** 中文国名 */
  name: string
  /** 中文全称 */
  full_name: string
  /** 国家类型 */
  countryType: 'EMPIRE' | 'REPUBLIC' | 'UNION' | 'COLONY' | 'KINGDOM' | 'SPLIT'
  /** 军事实力 (0-10) */
  military: number
  /** 工业能力 (0-10) */
  industry: number
  /** 人口/资源 (0-10) */
  population: number
  /** 对华威胁等级 (0-10) */
  threat: number
  /** 外交关系 */
  diplomacy: 'ALLIED' | 'FRIENDLY' | 'NEUTRAL' | 'HOSTILE' | 'WAR'
}
```

**国家类型说明：**

| 值 | 含义 |
|----|------|
| EMPIRE | 帝国（日本、英国） |
| REPUBLIC | 共和国（美国、法国） |
| UNION | 联盟/联邦（苏联） |
| COLONY | 殖民地（印度、越南） |
| KINGDOM | 王国（泰国、阿富汗） |
| SPLIT | 分裂/内战 |

**外交关系说明：**

| 值 | 含义 | 地图着色 |
|----|------|----------|
| ALLIED | 同盟 | 深蓝 `0x1a3a7e` |
| FRIENDLY | 友好 | 暗蓝灰 `0x2a3a5e` |
| NEUTRAL | 中立 | 暗橄榄 `0x2d3a2e` |
| HOSTILE | 敌对 | 暗红 `0x6b2020` |
| WAR | 交战中 | 深红 `0x4b0000` |

**AI 威胁评估公式：**

```
countryThreat = military * 3 + industry * 2 + population + diplomacyOffset
```

| 外交关系 | diplomacyOffset |
|----------|-----------------|
| ALLIED | -10 |
| FRIENDLY | -5 |
| NEUTRAL | 0 |

## 状态管理（useGameStore）

全局游戏状态统一由 Pinia store `src/stores/game.ts` 的 `useGameStore()` 管理，作为唯一真相源。

### 核心原则：事件溯源（Event Sourcing）+ Kernel 唯一写者

世界态（城市归属/驻军/士气、日期、势力存亡、战斗、玩家择势）**只能经事件 apply 变更**，状态是事件日志的纯函数结果：

- `applyEvent(e: GameEvent)` 是 **Kernel（唯一写者）**，放在 store 内（`game.ts`）。任何外部代码（组件 / AI / gameOrders）都**不得直接改世界态**，只能 submit 事件。
- 例外：开局 `initWorld()` 可直接覆盖式灌入种子（`chinaCities`），仅此一处直接写。
- 因为 reducer 是纯函数（无随机、无网络、无动画），**存档 = 序列化事件日志；读档 = 重放事件**——天然 replay 安全（详见下方「存档与事件溯源」）。

### GameEvent 类型（全部世界态变更）

| type | 字段 | 说明 |
|---|---|---|
| `capture` | `targetGb, actor, resultTroops?` | 占领/接收城市：易主 + 可选写入新驻军（k） |
| `attack` | `fromGb, targetGb, attackerLoss, defenderLoss` | 进攻未占：双方按裁定损耗扣兵（fromGb 为调兵源城） |
| `moraleChange` | `targetGb, delta` | 城市士气增减（0–100 钳制） |
| `produce` | `targetGb, amount` | 生产/征兵（驻军 +amount，k） |
| `moveTroops` | `fromGb, toGb, amount` | 城内调兵（fromGb 扣、toGb 加，单位 k） |
| `dateAdvance` | `date` | 推进全局日期（ISO 字符串） |
| `setFactionAlive` | `faction, alive` | 势力存亡（alive=false=灭亡，移出存活列表） |
| `battleStart` | `battleId, fromGb, targetGb, fromName, toName` | 战斗开始（登记元数据进 `battles`） |
| `battleEnd` | `battleId` | 战斗结束（移除元数据） |
| `selectFaction` | `faction, playerName` | 玩家择势（恢复玩家身份，replay 时靠它还原） |
| `narrative` | `playerInput, aiMessage` | 玩家自然语言指令 + AI 回话留痕（不改动世界态，仅存档） |

> ⚠️ `attack` / `produce` 两个事件类型已在 `applyEvent` 内实现 world 态变更，**但尚无对应 `order` 接通**（战斗减员、生产征兵尚未接入指令层）——目前它们不会被 `executeOrder` 触发，属于预留/休眠类型。不要把它们当作已可用的正常路径。

> **新增事件类型时**必须三处同步：① 在 `game.ts` 的 `GameEvent` 联合类型加分支；② 在 `applyEvent` 加对应处理；③ **同步更新 `initWorld()` 重置清单**（见下方存档章节），否则重复读档会脏数据。

### 读：getSnapshot（放 store 内）

AI 需要读取当前局势时，调用 `useGameStore().getSnapshot()`，返回一份**脱离响应式**的纯对象快照，
可直接 `JSON.stringify` 序列化后传给 LLM，且不会在调用方建立响应式依赖。

```ts
const state = useGameStore().getSnapshot()
// {
//   currentDate: string,
//   currentFaction: Owner | null,
//   activeFactions: Owner[],
//   ownership: Record<string, Owner>,        // gb -> 控制政权（派生自 cities）
//   cities: Record<string, { owner, troops, morale }>,  // 每城动态态
//   factionTroops: Record<string, number>,   // 势力总兵力（派生：所辖城市 troops 之和，k）
//   factionMorale: Record<string, number>,   // 势力士气（派生：按兵力加权的城市 morale 均值）
//   battles: BattleInfo[]
// }
```

> 注意：快照是拷贝，调用方对其修改不影响 store。

> 另：`stores/game.ts` 还暴露 `setPlayer(name, faction)`（设置玩家名 + 势力，择势早期用）、`snapshotForUndo()` / `restoreUndo(frame)`（撤销快照，调试/体验用），按需调用。

### 写：经 Kernel 的指令层 setter（gameOrders.ts）

AI / 指令入口回写世界态时，**统一经 `executeOrder({ order, ... })` 这一个入口**；带演出的指令函数（`arrowFly` / `radarPulse` / `orbBurst` / `battle` / `stopBattle` / `capture` / `moveTroops` 等）是 `gameOrders.ts` 的**内部实现**，仅由 `executeOrder` 调用，外部不得直接 import。它们**只负责画面（动画/镜头）**，世界态统一在 `executeOrder` 的 switch 尾端经 `applyEvent` 落地（先播动画、再改世界态）；`setFactionAlive` / `setCurrentDate` / `setCurrentFaction` 这类纯世界态指令则直接在 switch 内 `applyEvent`，无独立函数。

```ts
import { executeOrder } from '@/utils/gameOrders'

// 以下均经 executeOrder 分发，内部路由到对应 applyEvent：
await executeOrder({ order: 'capture', gb: '156450200', owner: Owner.JPN, resultTroops: 20 }) // → capture 事件：易主 + 写入新驻军（k）
await executeOrder({ order: 'setFactionAlive', faction: Owner.TIB, alive: false })            // → setFactionAlive 事件：灭亡
await executeOrder({ order: 'setCurrentDate', date: '1931-10-01' })                            // → dateAdvance 事件（会触发 autoSave）
await executeOrder({ order: 'setCurrentFaction', faction: Owner.KMT })                        // → store.selectFaction → selectFaction 事件
```

| 指令（order） | 参数 | 说明 |
|------|------|------|
| `capture` | `gb, owner, resultTroops?` | 占领/接收城市；结果路由到 `applyEvent` 的 `capture` 事件 |
| `setFactionAlive` | `faction, alive` | 设置势力存活，路由到 `setFactionAlive` 事件 |
| `setCurrentDate` | `date` | 推进全局日期，路由到 `dateAdvance` 事件（会触发 autoSave） |
| `setCurrentFaction` | `faction` | 选玩家势力，委托 `store.selectFaction` → `selectFaction` 事件 |

> 战斗生命周期不走这些 setter：`executeOrder({order:'battle'})` / `stopBattle` / `stopBattles` 内部直接调 `applyEvent({type:'battleStart'|'battleEnd'})` 管理 `battles` 元数据（动画句柄本地持有，不进 store）。

> 完整的指令词表（共 **13 条**：`arrowFly` / `radarPulse` / `orbBurst` / `battle` / `stopBattle` / `stopBattles` / `listBattles` / `fogCover` / `capture` / `moveTroops` / `setFactionAlive` / `setCurrentDate` / `setCurrentFaction`）及其 JSON 契约、三层可行性校验，见 `src/utils/aiOrderContract.ts`（`ORDER_TYPES` 为唯一真相源）；地点类参数全部经 `resolveLocationId` 解析（见上）。

### executeOrder 地点参数：名称 / GB 自动解析（resolveLocationId）

`executeOrder(json)` 是 AI 指令的 **JSON 协议统一分发入口**（`arrowFly` / `radarPulse` / `orbBurst` / `battle` / `capture` / `fogCover` / `stopBattle` 等 13 条，见 `ORDER_TYPES`）。其地点类参数（`from` / `to` / `gb`）**必须且已经过 `resolveLocationId()` 包一层解析再消费**，让 AI 既能传城市名也能传 gb 编码：

```ts
import { resolveLocationId } from '@/utils/locationResolver'

// executeOrder 内对每条地点参数做转换 + 判空
const fromId = resolveLocationId(json.from!)
if (!fromId) return { ok: false, reason: `城市不存在或拼写有误: ${json.from}` }
```

**`resolveLocationId(input)` 解析规则**（定义在 `src/utils/locationResolver.ts`）：

1. 先按自然语言地名查 `nameToId` 表（城市如「北平」、国家如「日本」）→ 命中返回对应 gb / iso_a3；
2. 查不到再当作 gb / iso_a3 直查（`resolveLocation`）→ 有效则返回原值；
3. 两者都不是 → 返回 `null`（调用方**必须判空**，不能直接透传给动画/相机，否则会崩）。

> 前提：name 空间（中文）与 id 空间（数字 gb / 字母 iso_a3）不相交，故「先查名再当 id」不会误吞合法 id。

**已套用该解析的指令与参数：**

| 指令 | 参数 | 说明 |
|------|------|------|
| `arrowFly` | `from`, `to` | 经 `resolveLocationId` 转换 |
| `radarPulse` | `from` | 经 `resolveLocationId` 转换 |
| `orbBurst` | `from`, `to` | 经 `resolveLocationId` 转换 |
| `battle` | `from`, `to` | 经 `resolveLocationId` 转换 |
| `moveTroops` | `from`, `to` | 经 `resolveLocationId` 转换 |
| `capture` | `gb` | 经 `resolveLocationId` 转换 |

> **新增约定（强制）**：今后在 `executeOrder` 里新增任何带地点参数的指令，`from` / `to` / `gb` / 任意地点字段都必须先经 `resolveLocationId()` 转换并判空，不得直接透传原始字符串，以保证「AI 传地名 / 传 gb 两条路等价、端到端可用」。校验层（`aiOrderContract.validateGameOrder`）同样走 `resolveLocationId` 判空，传地名会在校验阶段即被接受。

### 设计约定

- **读放 store、写经 Kernel**：读接口 `getSnapshot` 是 store action；写接口本质是 `applyEvent` 这个 reducer，外部通过 gameOrders 的薄封装提交事件，职责清晰。
- **城市态是唯一真相源**：`cities`（`shallowRef`，380 城全量 `CityState`）单源；`ownership` / `factionTroops` / `factionMorale` 均为派生，永不单独存储（兵力 = 城市级 `troops` 之和，士气 = 按兵力加权的城市 morale 均值）。
- **battles 仅存元数据**：`BattleInfo`（id/from/to/names/active），PixiJS 动画句柄留在 gameOrders 本地（`battleRegistry` / `activeBattles`），不进 store——因此动画句柄无法序列化，读档需重建（见下）。

## 引擎与 AI 调度架构

游戏引擎采用 **World System + Kernel（前端唯一写者）+ Agent-Kernel（前端调度）** 三层结构，后端 `apps/server` 只做 LLM 透传（完整设计见 `docs/引擎架构决策记录.md`）。本节只列已落地的关键模块，便于开发时按图索骥。

### 调度层（composables）

- `useGameScheduler.ts`：**回合/时间步调度器**。`advance()` 推进一个时间步——先让世界 AI 一次性生成该步的全部事件（含 `scheduledDate` / `stop` 标记），再按 `scheduledDate` 排序 `applyEvent` 逐条落地；遇 `stop`（或玩家相关事件）即暂停、交还玩家决策。读档成败、单条指令异常也在此收口并弹 Toast。
- `useAgentKernel.ts`：**Agent-Kernel 三段式对话调度**（玩家相关的跨阵营交互）。世界 AI 校验+路由 → 目标阵营 AI 回应 → 世界 AI 收口（产出 `narrative` + 结构化 `events[]` 走 `kernel.apply`）。玩家直接指挥己方走统一入口，不经对话。
- `useLlmClient.ts`：LLM 客户端封装（重试 / 超时 / 错误归一），是上面两者与后端 `/api/ai/chat` 之间的薄边界。

### AI 工具分层（utils）

玩家意图 → 指令的管线拆为四段，互不耦合：

- `aiClassify.ts`：玩家自然语言意图分类（指挥 / 询问 / 外交 …）；
- `aiContext.ts`：拼装发给 LLM 的上下文快照（世界态摘要 + 玩家势力）；
- `aiInvoke.ts`：统一 LLM 调用（流式 / 非流式）；
- `aiParse.ts`：从 LLM 输出抽取 JSON（容错 / 兜底）。

### 契约与校验（utils/aiOrderContract.ts）

- `validateGameOrder` / `validateOrders`：**结构校验**（城市码存在 / 枚举合法 / 日期格式），零 LLM 成本；
- `validatePlayerOrder` / `validatePlayerOrders`：玩家模式**战略校验**（`capture` 只能为本方、不能打自己人、`from` 须己方城等），禁用 `[setFactionAlive, setCurrentFaction]`；
- `validateFactionOrder` / `validateFactionOrders`：政权 AI 战略校验（actor 必为自身），额外禁用 `[setCurrentDate, setFactionAlive, setCurrentFaction]`；
- `WorldValidationResult` / `WarVerdict`（`feasible` | `impossible`）：世界 AI 对一批指令的两档可行性裁判（已取消"困难"档，2026-07-20 收敛）。

> 指令名称与 JSON 形状以 `aiOrderContract.ts` 导出的 `ORDER_TYPES`、`CONTRACT_SCHEMA_TEXT`、`PLAYER_AI_UNIFIED_PROMPT` 为唯一真相源，调试 AI 与生产 agent 共用同一份契约。

## 存档与事件溯源（localStorage）

事件日志 `eventLog` 是唯一持久化数据。存档 = 序列化事件日志；读档 = 重放事件。存档/读档接口均位于 store 内：

| 方法 | 参数 | 说明 |
|------|------|------|
| `save` | (slot, {label?}) | 把当前 `eventLog` + 玩家身份序列化到 localStorage 指定槽，返回是否成功 |
| `load` | (slot) | `initWorld` 重置 → 逐事件 `applyEvent` → 恢复玩家身份；返回是否成功 |
| `autoSave` | () | 写入 `auto` 槽，每次 `dateAdvance` 后自动触发（推进日期 = 回合边界） |
| `listSaves` | () | 列出所有槽位摘要 `slot → {label, savedAt, currentDate, eventCount}` |
| `deleteSave` | (slot) | 删除指定槽位 |

- **多槽位 + 每回合自动存**：key 前缀 `mapwar_save_`；`auto` 槽由 `autoSave()` 在每次 `dateAdvance` 后写入。
- **replay 流程**：`load(slot)` = `isReplaying=true` → `initWorld()`（重置基线）→ `for (e of eventLog) applyEvent(e)` → 直接赋值完整 `eventLog` → 返回。**玩家身份由日志中的 `selectFaction` 事件在 replay 中自动恢复**，无需单独存。
- **`isReplaying` 守卫**：replay 期间 `applyEvent` 不重复追加日志、不触发 `autoSave`；地图归属 watcher 也跳过重绘（改由 `loadTest` 手动调 `loadLayer`）。**replay 结束的 `isReplaying=false` 必须由调用方（如 `loadTest`）显式复位**，store 内部不在 `load` 末尾自动复位。
- **`cities` 用 `shallowRef`**（关键性能决策）：380 城若用 `reactive` 会在 `initWorld` 逐条赋值时触发数百次响应式副作用（DOM / 动画崩溃）。故 `cities` 用 `shallowRef` + **一次性整体赋值**（`cities.value = seed`），城市态变更后 `triggerRef(cities)` 手动通知。新增需要响应的世界态字段时，同样避免逐条 reactive 写入。
- **读档后动画恢复**：战斗动画句柄不进 store（PixiJS 句柄无法序列化）。`loadTest` 在复位 `isReplaying` 前调 `gameOrders.restoreActiveAnimations()`：扫描 `store.battles` 中 `active=true` 的条目，重建 PixiJS 动画并注册到模块内部映射。**新增需持久化的运行时动画，统一用「元数据进 store + 句柄本地持有 + 读档时重建」模式**。

### initWorld 重置清单（⚠️ 新增世界态字段必须同步更新）

读档前所有受 `applyEvent` 管理的状态字段必须被 `initWorld()` 重置为开局基线，否则重复读档会累积脏数据（已踩坑：`battles` 未重置导致重复读档战斗条目翻倍）。

| 字段 | 重置值 |
|---|---|
| `cities` | 从 `chinaCities` 覆盖式灌入（一次性 `cities.value = seed`，不删旧 key） |
| `currentDate` | `'1931-04-01'` |
| `activeFactions` | 10 个初始势力 |
| `eventLog` | `[]` |
| `battles` | `[]`（否则重复读档战斗条目翻倍） |
| `playerName` | `''` |
| `currentFaction` | `null` |

## 轻量提示（Toast）

羊皮纸风轻提示系统（顶部居中、滑入淡出），由「队列 + 容器」两段式构成，决定弹什么与怎么显示解耦。

### 队列：`composables/useToast.ts`（模块级单例）

`useToast()` 返回 `{ toasts, push, dismiss, clear }`；`toasts` 是响应式数组（`<ToastStack>` 订阅渲染），所有提示（指令结果、时间推进、读档成败）都经它入队。

```ts
import { useToast } from '@/composables/useToast'

// 入队一条提示，返回 id（可手动 dismiss）
useToast().push({
  icon: 'sword',      // tabler 图标名（见 GameToast 图标注册表），如 'sword'/'flag'/'clock'
  title: '出兵',      // 主标题（粗体）
  text: '北平 ⇢ 天津', // 副文案
  tone: 'amber',      // 色调，决定左边框/图标色
  duration: 3400,     // 自动消失毫秒；0 = 常驻
})
```

- `toasts` 上限 `MAX_STACK = 5`，超出自动挤掉最旧一条，防刷屏；
- `duration` 默认 `3400`ms，`0` 表示常驻（需手动关闭）；
- 天然 replay 安全：队列只在活操作里被 `push`，读档重放（`applyEvent` 直跑）不会触发。

### 色调 `ToastTone`

`cinnabar`（宣战/开战/占领）· `amber`（出兵进攻）· `blue`（侦察）· `green`（成功/读档成功）· `purple`（势力参战）· `neutral`（停战/删档/日期）· `error`（失败/覆灭/异常）。

### 渲染：`components/ui/GameToast.vue` + `ToastStack.vue`

- `GameToast.vue`：单条羊皮纸卡片（左边框 + 图标按 `tone` 着色，图标经字符串名→组件注册表解析）；
- `ToastStack.vue`：`<Teleport to="body">` 容器，顶部居中、日期条下方（`top: 64px`，`z-index` 9000），用 `TransitionGroup` 做滑入淡出；
- `App.vue` 已挂载 `<ToastStack />`。

### 触发点（单点收敛）

- 指令结果：`executeOrder` 尾端统一 `popToast(order, result)`（成功/失败都弹，文案取 AI 的 `order.text` 与城市名）；
- 时间推进：`playTimeJump()` 完成处弹（`neutral` 日期条）；
- 读档成败/删档：`useSaveGame.ts`；
- 交还玩家 / `needsPlayerDecision`、单条指令异常：`useGameScheduler.ts` 的 `advance()`；
- 择势：`stores/game.ts` 的 `selectFaction`（`!isReplaying` 守卫，覆盖开局与 god-mode 两条路径）。

## 动画函数库

文件位置：`apps/web/src/utils/troopAnimation.ts`（纯 PixiJS，与 Vue 解耦）。

> ⚠️ **地点参数用「地点 id」，不是屏幕坐标。** 所有动画函数的起点/终点都是**地点 id**（`fromId` / `toId` / `targetId`），坐标由函数内部经 `resolveLocationXY()` 从地点注册表查出（城市用 gb 编码、国家用 iso_a3）。不要把 `{x, y}` 屏幕坐标传进去——那是旧接口，已废弃。`resolveLocationXY` 返回 `null` 时会 `console.warn` 并安全跳过，不会崩。

### playArcAnimation - 弧线动画（通用）

从起点到终点的弧线动画，支持两种模式和文字弹出。

**模式：**
- `mode: 'dots'` - 多箭头沿弧线移动（适合派兵、行军）
- `mode: 'orb'` - 光球沿弧线移动（适合宣战、施法）

**效果组合：**
- `explosion: true` - 到达后爆炸+冲击波
- `text: '文字'` - 到达后在目的地上方弹出飘字


## 云雾蒙太奇（Cloud Transition）

全屏水墨云雾「盖住 → 停顿 → 散开」的蒙太奇演出，参考海岛奇兵的云雾切换。
所有渲染代码集中在 `src/utils/cloudTransition.ts`（**纯 PixiJS 渲染原语，与 Vue 解耦，没有独立的 Vue 组件**）。

### 渲染层与全屏覆盖

- 云雾挂在 `app.stage` 顶层（`screen` 坐标系，不受相机平移/缩放影响），永远稳稳盖住整张地图视口。
- 云雾是画在 PixiJS **canvas 内部**的位图。canvas 之上用 `position:absolute` + `z-index` 浮着的 Vue UI（按钮、面板、弹窗、右侧 HUD）**物理上无法被 canvas 内画的云雾盖住**——这是浏览器渲染模型限制，不是层级 bug。
- 因此演出采用「视觉隐藏」方案：云雾盖住地图的同时，把那些顶层 UI 同步淡出，整屏看上去就被雾吞没了。具体机制见下方「新组件兼容云雾隐藏」。

### 触发方式一：调试面板（手动）

地图界面左上「调试」→「云雾切换测试」按钮，即调用一次默认演出（实现在 `LeafletMap.playCloudTest`）。

### 触发方式二：代码里直接调用（推荐给引擎/演出层复用）

```ts
import { playCloudTransition, disposeCloudTransition } from '@/utils/cloudTransition'

// 必须传入已 init 的 PixiJS Application
await playCloudTransition(app, {
  coverDuration: 1300,   // 四边汇聚盖满（ms）
  holdDuration: 800,     // 停顿（ms）
  revealDuration: 1300,  // 退散揭开（ms）
  onMidpoint: () => {    // 盖满瞬间触发，藏一次状态切换（时间流逝蒙太奇）
    useGameStore().applyEvent({ type: 'dateAdvance', date: '1931-11-01' })
  },
})
```

- `playCloudTransition` 返回 `Promise`，演完 resolve；`playing` 期间重复调用会被忽略（防重叠）。
- 演出期间组件自行处理拉幕/散幕；相机锁定由调用方负责（见下方指令层）。
- 组件卸载 / 场景销毁时调用 `disposeCloudTransition()` 紧急清理（已挂到 `LeafletMap.onUnmounted`，防止 `cloud-active` 状态类卡住 UI 隐藏）。

| 选项 | 类型 | 说明 |
|------|------|------|
| coverDuration | number (ms) | 四边汇聚盖满时长，默认 1300 |
| holdDuration | number (ms) | 中段停顿时长，默认 800 |
| revealDuration | number (ms) | 退散揭开时长，默认 1300 |
| onMidpoint | () => void \| Promise<void> | 盖满瞬间触发一次，用于藏状态切换；失败不影响演出 |

### 触发方式三：AI 指令层 / 时间跳跃统一入口

游戏指令层 `src/utils/gameOrders.ts` 提供了时间跳跃的**统一公共入口 `playTimeJump(date)`**，它内部用云雾蒙太奇裹住一次 `dateAdvance`（盖满瞬间 `applyEvent({type:'dateAdvance'})`，揭开后玩家看到新日期），并推送「时间推进」提示。推荐引擎/调度层一律走它推进日期：

```ts
import { playTimeJump, executeOrder } from '@/utils/gameOrders'

// 时间跳跃（云雾蒙太奇 + 推进日期，onMidpoint 改钟），推荐入口
await playTimeJump('1931-11-01')

// 仅播云雾、不推进日期（无参数指令 fogCover）
await executeOrder({ order: 'fogCover' })
```

注意：

- `init` 必须传入第三个参数 `app`（PixiJS Application）：`init(worldContainer, cameraController, app)`。未注入 app 时 `playTimeJump` 自动降级为直接 `applyEvent({type:'dateAdvance'})`，不抛错。
- `fogCover` 指令始终播放云雾蒙太奇（内部 `await playCloudTransition`）。它**不推进日期**，仅作视觉演出；需要「盖住时藏一次状态切换」请直接用 `playTimeJump`（已内置 `onMidpoint` 改钟）。
- `executeOrder` 的 `fogCover` 分支无法携带函数回调（JSON 协议不传函数）；若要挂自定义 `onMidpoint`，直接调用 `playCloudTransition(app, opts)`（见「触发方式二」）。
- 旧文档曾写的 `cloudTransition` / `executeOrder({order:'cloud'})` 已不存在：指令改名为 `fogCover`，中段逻辑统一收敛到 `playTimeJump`。

### 新组件兼容「云雾隐藏」（重要）

演出开始会往 `<body>` 加 `cloud-active` 类，全局 CSS 负责淡出所有带 `map-ui` 类的元素：

```css
/* App.vue 全局样式 */
.map-ui { transition: opacity 0.45s ease; }
body.cloud-active .map-ui { opacity: 0; pointer-events: none; }
```

**新增任何浮在地图之上的 UI 组件（按钮、面板、弹窗、HUD、图例、声明条等），根元素必须加 `map-ui` class**，否则演出时云雾盖住了地图、该组件却仍浮在云上，破坏「整屏被雾吞没」的观感。

要点：

- 普通组件：在模板浮动根元素写 `class="map-ui"`（多根组件需包一个 wrapper，或显式绑定到浮动根）。
- 使用 `<Teleport to="body">` 的弹窗（如 `GameModal`）：根 `.modal-overlay` 会被挂到 body 下，选择器 `body.cloud-active .map-ui` 仍能命中，加 `map-ui` 即可。
- 若组件自身已有 `:class` 绑定（如条件激活态），用 `:class="['map-ui', ...]"` 合并，不要覆盖。
- 已加 `map-ui` 的组件清单：`.layer-switcher`、`<GameDateDisplay>`、`<GameContextMenu>`、4 个 `<GameModal>`（调试/AI/战斗/免责）、`<LegendPanel>`、`.disclaimer-bar`、`<PlayerStatusPanel>`、`<ToastStack>`（Teleport 到 body，云雾期间同步淡出）。

## 开发环境与运行脚本

本项目是 pnpm 工作区（根 `pnpm-workspace.yaml`），前端 `apps/web`、后端 `apps/server` 各自独立。

### 安装与启动

```bash
pnpm install                        # 根目录安装全部 workspace 依赖
pnpm --filter map-war-web dev       # 启动前端（Vite，默认端口 5714）
pnpm --filter map-war-server start  # 启动后端（NestJS，默认端口 3001）
```

> ⚠️ **前后端端口必须对齐，否则本地联调被 CORS 拦截。** 前端 Vite 开发端口是 **5714**，后端 `CORS_ORIGIN` 默认 **5173**——两者不一致时浏览器会拒绝 `/api` 请求。本地联调请把 `apps/server/.env` 的 `CORS_ORIGIN` 改成 `http://localhost:5714`（或统一两边端口）。

### 前后端对接

- 前端通过 Vite 代理把 `/api` 转发到 `http://localhost:3001`（`apps/web/vite.config.ts` 的 `server.proxy`），所以前端代码里写 `/api/ai/chat` 即可，不用硬编码后端地址。
- 后端 `POST /api/ai/chat` 是 **OpenAI 兼容的透传代理**（详见 `AI底层接口封装方案.md`）：原样转发请求体、返回 `ChatCompletion` JSON；API Key 只在 `apps/server/.env`，前端零硬编码。
