# 架构设计

> 本文档记录 map-war 项目在「AI 驱动战争推演」方向上的**已确定架构方向**。
> 它是设计讨论的收敛结果，供后续实现对照。标注「待定」的部分需进一步拍板。
> 每次讨论收敛后更新本文档，保持单一事实来源。

---

## 1. 产品愿景与核心理念

- 以 **1931 年九一八事变前后**为背景的中国战争策略游戏。
- **玩家 = 政权领袖**：进行指挥 → AI 回复并操作 → 计为一轮；玩家无操作时可**快进**，由 AI 自动推演战局。
- 系统可向 AI **喂数据**（当前游戏状态），AI 只负责**产出下一步意图**，不直接改写世界。
- 目标：让多个 AI 势力（及玩家）在同一套规则下自主演化，支持「人机对战 / 全 AI 推演 / 逐步喂数据观察」多种用法。

## 2. 已确定的架构方向

### 2.1 时间模型：无回合制，全局日期时钟

- **不使用回合（turn）**，改用全局日期 `currentDate`（如 `"1931-09-18"`）。
- **玩家操作零耗时**：玩家（或任一势力）的指挥只生成「未来事件」，**不推进时钟**。
- 时钟只随**事件被 Kernel 应用**而推进到该事件的 `scheduledDate`。

### 2.2 事件溯源 + 单一写者（Kernel 模式）

- 所有行动（玩家、势力 AI、世界 AI）**只提交事件（Event），不直接修改世界状态**。
- **世界状态只能由 Kernel 修改**（单一写者），杜绝并发写冲突。
- Kernel 维护**事件队列**，按 `scheduledDate` 排序的时间线。
- Kernel 应用（apply）一个事件时：
  1. 校验世界状态（目标仍属敌方？条件满足？）
  2. 真正修改权威状态（如 `capture` 改 `ownership`）
  3. **触发动画作为副作用**（播放派兵/宣战/战斗动画，属展示层）
  4. 将 `currentDate` 推进到该事件日期
- **快进（fast-forward）= 持续 `advance()` 执行队列，直到遇到「重要事件」才停止，把控制权交还玩家**。

### 2.3 Agent-Kernel 的多 AI 架构

- **每个主要势力配一个 AI 实例**，另设一个**世界 AI**（负责全局叙事事件、各势力操作汇总）。
- 采用 **Agent-Kernel 模式**：由一个 Kernel 统一调度各 Agent。
- **智能动态增减 AI**：势力灭亡 → 移除其 AI；新势力崛起 → 新增 AI。
- **AI 永远只提交事件，绝不触碰世界状态**（与 2.2 一致）。

### 2.4 玩家身份：开局选择政权

- 进入游戏时，玩家从 10 个政权中选择一个作为自己的势力（如 KMT / JPN / CCP …）。
- 选择结果存入 **Pinia game store 的 `currentFaction`**。
- `currentFaction` 用途：
  - UI 渲染：高亮「我方」城市、区分敌我着色。
  - 玩家视角的 `getState()` 默认参数。
- 快进 / Kernel 循环内，以敌方势力身份调用 `getState(faction)` 切视角喂给对应 AI。

### 2.5 状态存储：Pinia game store（`useGameStore`）

新增独立游戏状态 store（当前 `stores/counter.js` 仅为空模板，需新建/替换）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `currentFaction` | `string \| null` | 玩家选中势力；开局 `/select` 设定 |
| `currentDate` | `string` | 全局日期时钟（见 2.1） |
| `ownership` | `Map<gb, owner>` | **可变**城市控制权，初始化自 `chinaCities` 的 `owner` |
| `eventQueue` | `Event[]` | Kernel 维护的事件时间线（按 `scheduledDate` 排序） |
| `activeFactions` | `string[]` | 当前存活/有 AI 的势力（动态增减） |
| `battles` | `Battle[]` | 当前进行中的战斗（沿用现有 battleRegistry 思路） |

> 渲染层（LeafletMap）需改为从 store 读取 `ownership`，而非直接读静态 `chinaCities`。

### 2.6 `getState` 接口（AI 感知入口）

请求：
```json
{ "order": "getState", "faction": "KMT" }
```
`faction` 可选，默认取 `currentFaction`（玩家视角）；快进循环内传敌方势力切视角。

响应（候选形态，字段已定方向）：
```jsonc
{
  "ok": true,
  "currentDate": "1931-09-18",        // 全局时钟（替代原 turn）
  "myFaction": "KMT",                 // = faction 或 currentFaction
  "ownership": { "156500000": "KMT", "156450200": "JPN" }, // 可变归属，紧凑 Map
  "myCities": ["156500000", "..."],   // 由 ownership 过滤出的我方城市 id
  "battles": [                        // 当前进行中战斗
    { "id": "battle_1", "from": "...", "to": "...", "fromName": "...", "toName": "..." }
  ],
  "pendingEvents": [ "..." ],         // 事件队列中待发生事件
  "lastEvents": [ "..." ],            // 上一帧已发生事件（玩家关注点）
  "activeFactions": ["KMT", "JPN", "CCP", "..."] // 当前有 AI 的存活势力
}
```

### 2.7 `gameOrders.js` 重构方向

- 现状：直接调用动画 / 隐含改状态（attack/battle/declareWar/scout）。
- 新定位：降级为 **「事件提交 API」**——AI/玩家调用它 = 向 Kernel 投递事件意图。
- **动画改为 Kernel apply 事件时的副作用**，而非 AI 直接触发。
- 现有 `listBattles` / `stopBattle` / `stopBattles` 可保留（战斗运行时管理），其余指令需包裹为事件提交。

## 3. 关键数据结构草案

### 3.1 事件 Event（AI/玩家只产出此对象）

```jsonc
{
  "id": "evt_0001",                         // Kernel 分配
  "type": "capture|declareWar|battle|scout|move|produce|diplomacy|...",
  "actor": "JPN",                           // 发起势力
  "target": "156450200",                    // 目标地点 id（可选）
  "scheduledDate": "1931-09-18",            // 计划执行日期
  "important": true,                        // 是否重要（快进停止条件）
  "intent": { "from": "156500000", "to": "156450200", "text": "..." }, // 具体参数
  "status": "pending|applied|failed|cancelled"
}
```

### 3.2 Kernel 职责（唯一写者）

- `submit(event)`：校验 actor 合法性 → 入队（按 `scheduledDate` 排序）→ 返回 event id
- `advance()`：取队首 → 校验世界状态 → apply（改 store + 动画副作用 + 推进 `currentDate`）→ 若 `important` 则停（交还玩家）
- `fastForward()`：`while 队首不 important: advance()`
- 所有 state 修改只经 Kernel

### 3.3 玩家循环与快进的统一

```
loop:
  state = getState()                                   # 全局 + 各势力视图
  for f in 活跃势力(除玩家): ops[f] = f的AI.decide(getState(f))   # 各势力 AI 出事件
  worldEvent = 世界AI.decide(getState())                # 世界 AI 出「下一事件 + 时间点」
  for evt in ops + [worldEvent]: kernel.submit(evt)     # 全部提交，不直接执行
  kernel.fastForward()                                  # 执行到重要事件（涉及玩家）停止
  → 交还玩家手动指挥；玩家提交后再次进入 loop
```

> 玩家「指挥→AI回复=一回合」本质 = 玩家提供自己势力的事件 → 系统跑 AI 循环直到下一个与玩家相关的事件。两套时间模型（玩家回合 / AI 事件）被此循环统一。

## 4. 待定问题（需后续拍板）

| # | 问题 | 选项 / 备注 |
|---|------|------------|
| Q1 | **「重要事件」判定规则** | (A) 类型白名单（capture/declareWar/灭亡=重要）；(B) 是否涉及玩家势力（推荐）。还须定：涉及即停，还是仅「需玩家决策」才停。 |
| Q2 | **玩家军事行动是否带行军延迟** | `scheduledDate = 当前日`（立即执行，契合「一回合」）/ `当前日 + N天`（战线更真实，但回合边界模糊）。 |
| Q3 | **事件 `type` 体系** | 扩展原 gameOrders order 枚举 / 新建事件类型体系（推荐，原指令作为事件 `type` 子集）。`AI指定.md` 需随之重写。 |
| Q4 | **同日期多事件的执行顺序与依赖** | 按提交顺序 / 势力优先级；依赖不满足则事件 `failed`。 |
| Q5 | **「主要势力」定义与动态增减依据** | 哪几个算主要？JPN（外部势力）算不算？势力灭亡删 AI、崛起加 AI 的具体触发条件。 |

## 5. 实施路线（建议）

1. **政权选择（下一步）**：新增 `/select` 路由 + 选政权界面（用 `OWNER_LABELS`/`OWNER_COLORS` + `1931设定.md` 文案）；选中后写 `store.currentFaction`；`/` 路由拦截未选时跳转 `/select`。
2. **Pinia game store + 可变 ownership**：建 `useGameStore`，初始化 `ownership`（自 `chinaCities`）、`currentDate`、`currentFaction`；渲染层改读 store。
3. **getState 基础版**：基于 store 输出 `currentDate / myFaction / ownership / myCities / battles`。
4. **Kernel + 事件队列 + capture/resolve**：实现 `submit/advance/fastForward`，把 `capture` 等真正落到 `ownership`；动画作为 apply 副作用。
5. **Agent-Kernel 循环 + 快进 UI**：各势力 AI 出事件、世界 AI 出下一事件、快进按钮驱动 `fastForward()`。
6. **世界 AI / 事件生成模型**：定义「AI 返回下一事件时间点 + 各方操作」的提示词与解析。
