# 游戏引擎架构决策记录（World System / Kernel / Agent-Kernel）

> 整理自 2026-07-11 与用户的 grill-me 深度讨论。
> 本文只记录讨论中**拍板的设计结论**，不含任何代码实现细节（实现后做）。

---

## 0. 起点：代码与文档的落差（讨论前的现状）

讨论前通读代码与文档，确认原 `架构设计.md`、`AI指令.md` 写得很美，但落地程度很低：

- 世界状态 store 存了 `ownership / currentDate / currentFaction / activeFactions / battles` 五项，但**没有事件队列**——而事件队列是 Kernel 的命脉。
- AI 写回接口是直接改 store 的 setter，**违反"世界状态只能由 Kernel 修改"** 的原则。
- `AI指令.md` 里的指令全是动画（`attack/scout/declareWar/battle`），**没有一个能改变世界态**（`capture` 不在协议里）。即 AI 连"占领城市"都做不到，只能放烟花。
- AI 服务端只是一次模型透传；**没有 Agent 抽象、没有调度器、没有世界 AI、没有快进循环**。

---

## 1. 核心定位（先拍根，再谈细节）

| 层 | 定位 | 权威态位置 |
|---|---|---|
| **World System** | 世界状态聚合与真相源 | 前端 |
| **Kernel** | 唯一写者（reducer 模式），世界态只能经事件 apply 变更 | 前端 |
| **Agent-Kernel** | 前端调度 + 世界 AI / 阵营 AI 编排 | 前端 |

**权威态放前端，后端只做 AI 转发代理**（模型透传），不放任何游戏逻辑。

---

## 2. 逐条拍板的决策（按讨论顺序）

### 2.1 Kernel = 前端唯一写者（reducer 模式）
- 世界状态只能是"事件 apply"的纯函数结果。
- 删除直接改 store 的后门 setter，改为"事件提交"入口；世界态只能经 Kernel 的 reducer 变更。
- store 对外只读，唯一的状态修改入口不暴露给组件/AI，只有 Kernel 能触达。
- 动画作为 apply 的副作用，经事件总线解耦，仅在非快进时触发。

### 2.2 快进语义
- **快进 ≠ 跳过动画。** 快进（长距离跳跃段）是「应用队列中每个事件（改状态、推进日期）」+ 末尾播**时间流逝蒙太奇**——蒙太奇本身就是一段演出，不是无动画。
- 区别：普通推进逐事件播放动画；快进段**跳过逐事件动画**，改由一次蒙太奇压缩演出（视觉上仍有动画，只是不逐个播）。
- 因此"先播动画、再提交状态"在任何模式下都成立：普通推进播逐事件动画后提交；快进段由蒙太奇统一演出，逐事件动画被省略但状态照常提交。

### 2.3 Agent-Kernel = 前端调度，顺序流
- **每步只请求"涉及到的势力 AI + 世界 AI"**，非全势力并行（原"所有势力每帧并行决策"的循环作废）。
- 各势力**自主产出自己的事件**（actor 必为自身）。
- 世界 AI **不当"总导演"**，而是**校验者 + 叙事者**：检测非法/不合理指令并"打回"（例：玩家扮演川系却下令"让蒋介石派兵打美国"——不能指挥他国、目标不合理，应被拒），并生成叙事文案。

### 2.4 外交/跨阵营交互 = 消息总线层（叠加在 Kernel 之上，非取代）
玩家发起的跨阵营请求走三段式：
```
玩家自然语言 → 世界AI(校验+路由) → 目标阵营AI回应 → 世界AI收口
```
- **世界 AI 收口须同时产出叙事文案 + 结构化事件**（走 Kernel apply），否则外交不改世界态（回到"AI 只会放烟花"）。
- 多轮以世界 AI 的"完成"标志终止；拒绝也算完成（不出事件）。
- 代码约定：玩家 = 川军（`SCC`），目标晋系 = `SHX`（`data/owners.ts`）。

### 2.5 两类事件生产者（Agent-Kernel vs 世界 AI）
1. **Agent-Kernel（专属对话）**：仅限**与玩家直接相关**的阵营（玩家交战的、或玩家使团/外交目标），走 §2.4 三段式对话产出事件。
2. **世界 AI（一次性批量）**：其余所有阵营事件由世界 AI 据当前局势**一次性生成一批**（带日期），不经逐阵营对话。
- 成本收敛：平时每回合仅 **1 次世界 AI 批量 + 玩家相关对话的 2~3 次**。

### 2.6 玩家动作按"反馈时效"分流 + 推进/停止
- **能及时反馈的**（电报/使团等跨阵营交互）→ 世界 AI 判断后路由到对应方走 Agent-Kernel 对话，交涉完成后世界 AI 再补一批背景事件（如某系向某系宣战）。
- **不能及时反馈的**（派人访苏联等长距离/长周期）→ 世界 AI 直接批量生成该时段所有事件并推进：
  - 无"玩家相关事件"则播放时间流逝蒙太奇并跳到抵达日；
  - 期间若生成玩家相关事件（如某 AI 对玩家宣战）则停在该时刻交还玩家。
- 推进 = 按日期排序 apply 批次，遇世界 AI 标注"需玩家决策"的事件即停（交还玩家；无则跑完批次正常结束回合）。
- 重大意义：原未决的"调度器选 actor""快进期间调不调 LLM"被此模型**直接消解**——无独立调度器（世界 AI 批量生成），推进期不调 LLM 只 apply。

### 2.7 成本收敛（混合 AI）
- 次要/小势力用基于规则的 AI（`attackScore` 启发式）兜底，避免每次都调模型。
- 仅玩家势力 + 主要势力用大模型 AI。
- 快照瘦身：别每步都喂全量城市，只喂周边若干跳 + 与上帧的差异。

### 2.8 动画语义（纠正）
- **普通推进**（玩家决策后世界 AI 补的批次，按日期逐事件 apply）**逐事件播放动画**（派兵动画、建交动画等）。
- **仅长距离跳跃段**（不及时反馈的长周期行动）跳过逐事件动画、末尾播"时间流逝"蒙太奇。
- 原"快进跳过动画"现特指长距离跳跃段，不泛指所有推进。

### 2.9 reducer → 事件溯源 / replay 天然成立
- 状态是事件日志的纯函数 → 状态重放是确定性的，不需再调任何 AI。
- 存档 = 序列化事件日志；replay = 重放事件结果，不重放 AI 决策。
- **已落地（2026-07-12）**：`eventLog` 持久化到 localStorage（多槽位 + `auto` 槽每回合 `dateAdvance` 自动存），`load(slot)` 走 `initWorld()` 重置 → 逐事件 `applyEvent` 重放 → 由日志内 `selectFaction` 事件恢复玩家身份。具体接口、replay 流程、`isReplaying` 守卫、`cities` 用 `shallowRef` 一次性赋值、读档后 `restoreActiveAnimations()` 重建战斗动画，见 `开发文档.md` 的「状态管理」「存档与事件溯源」两节。
- 前向坑（记事，后做）：reducer 须保持纯 + 向后兼容，事件带 schema 版本号。

**replay 前置检查清单（⚠️ 新增 store 字段时必须更新）**

读档流程：`initWorld() 重置基线` → `for each event in save: applyEvent(event)` → 完成（玩家身份由 selectFaction 事件在 replay 中恢复）。

`initWorld()` 必须保证读档前所有受 `applyEvent` 管理的状态字段被重置为开局基线，否则重复读档会累积重复数据（已踩坑：`battles` 未重置导致重复读档时战斗条目翻倍）。

当前 `initWorld()` 重置清单：

| 字段 | 重置值 |
|---|---|
| `cities` | 从 `chinaCities` 覆盖式灌入（一次性 `cities.value = seed`，不删旧 key，种子有全部 380 城） |
| `currentDate` | `'1931-09-18'` |
| `activeFactions` | 10 个初始势力 |
| `eventLog` | `[]` |
| `battles` | `[]` |
| `playerName` | `''` |
| `currentFaction` | `null` |

**新增 store 中的世界态字段时，必须同步在 `initWorld()` 中加上重置逻辑**，否则 replay 不干净。

## 3. 收尾三项拍板（22:40 后）

| # | 问题 | 决策 |
|---|---|---|
| ① | "是否需玩家决策"由谁决定 | 由**世界 AI 在生成事件时直接标注**（引擎读标志即可，不需代码谓词，玩家无需定义规则） |
| ② | 长距离时段长度/抵达日由谁算 | 由**世界 AI 估算**（游戏不需精确） |
| ③ | 玩家直接指挥己方阵营（出兵/生产）走哪条路 | 走 **统一入口**——所有玩家自然语言先过世界 AI（校验+即时出事件，己方指令不拉对话） |
| ④ | 动画边界 | 普通推进**播逐事件动画**，仅长距离跳跃段跳过（末尾蒙太奇）——纠正后 |

### 22:46 收尾三项拍板
1. **"需玩家决策"事件的体验**：自由文本为主、世界 AI 给 1~2 个建议选项为辅。
2. **世界 AI 返回内容坏了怎么办**：解析失败重新请求，超过 **3 次失败**则抛出错误。
3. **多轮对话的终止**：**不设轮次上限**（玩家一直重复由他自便）。

## 4. 最终收敛的玩家回合循环（设计描述）

```
玩家自然语言 ──(统一入口)──► 世界AI 规划
        │
        ├─ 及时反馈(电报/使团) ─► 路由到对应方 ─► 目标阵营AI 回应 ─► 世界AI 收口(双件套：叙事+事件)
        │
        ├─ 长距离(访苏联) ─► 世界AI 批量生成该时段事件(静默段 + 时间流逝蒙太奇)
        │
        └─ 己方指令(出兵/生产) ─► 世界AI 即时出事件(不拉对话)
        │
        ▼
世界AI 补充 ── 补背景事件(某系宣战 / 某系建交 …)
        │
        ▼
事件入队 ── 按日期排序
        │
        ▼
推进 ── 逐事件 apply + 播动画，遇"需玩家决策"事件则交还玩家
```

## 6. 数据模型订正：兵力存于城市（2026-07-12）

- **撤销**原"势力总兵力为权威存储、不存每城驻军"的结论（该结论源于 23:57 的量化 grill，模型选错维度）。
- **新模型**：兵力是**城市级**属性（`chinaCities.seed.json` 每城 `troops`，单位 千/k）；**势力总兵力 = 所辖城市 `troops` 之和（派生，不存储）**。士气同为**城市级**（`chinaCities.seed.json` 每城 `morale` 0–100）；**势力士气 = 按兵力加权的各城 morale 均值（派生）**。
  - 这同时证明最初"把初始值写进 `chinaCities.seed.json`"的直觉是对的——兵力本就该落在城市文件。
  - `factionSeed.ts` 整文件取消（与用户"这个不用吧"一致；`factionSeed.ts` 议题本身已撤销）。
- **攻城结算**：`attack` 带源城/目标城/投入 `troops` → 世界 AI 裁定占/败 + 损耗，结果写进事件 → Kernel.apply 确定性执行（replay 安全，对应 §2.9）。
- 详细数据模型见 `架构设计.md` §3.4。
- **落地（2026-07-12）**：seed → store 单向灌溉已实现。`useGameStore().cities` 为唯一真相源（store 持 380 城全量 `CityState`），`ownership` 改为派生计算属性；`applyEvent` 为唯一写者（`gameOrders.captureCity` 已路由到 `applyEvent` 的 `capture` 事件）；`getSnapshot()` 扩展每城 `troops/morale` + 势力派生聚合；信息面板/地图标记均改读 store。详见 `架构设计.md` §3.5。
