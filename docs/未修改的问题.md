## 🟡 值得做但需多想想

### 1. `ChatEntry` 同名不同形

| 文件 | 形状 |
|---|---|
| `components/AdvisorPanel.vue` ~L96 | `{ user, reply, suggestions? }` |
| `components/PlayerAiPanel.vue` ~L162 | `{ user, msg, orders[], rejected?[], impossible?[], validationSummary? }` |

两个组件都定义了局部 `interface ChatEntry`，但语义不同——AdvisorPanel 侧重问答对话，PlayerAiPanel 侧重指令校验结果。不建议强行合并为一个类型（会让所有字段都变 `?` 可选，反而降低类型安全）。

**建议：重命名区分。** `AdvisorChatEntry` / `PlayerChatEntry`，或提取共同基类型 `interface ChatEntryBase { user: string }`。

### 2. `UnifiedAiResponse` 近重复（verdict 精度不同）

| 文件 | `results` 元素类型 |
|---|---|
| `utils/aiParse.ts` ~L8 | `verdict: string`（宽类型，不依赖 `WarVerdict`） |
| `composables/useAiDebug.ts` ~L55 | `verdict: WarVerdict`（精确类型，`'feasible' \| 'impossible'`） |

`aiParse.ts` 是纯工具层，故意用宽松类型以保持零依赖；`useAiDebug.ts` 需要精确类型来做业务逻辑。

**建议：** `aiParse.ts` 保持泛用版本，`useAiDebug.ts` 定义 `UserModeAiResponse extends` 或用交叉类型覆盖 `results` 字段类型。不强行合并。

### 3. `{ role: string; content: string }[]` 聊天消息数组三处内联

| 文件 | 位置 |
|---|---|
| `composables/useAiChat.ts` ~L4 | `AiChatRequest.messages` |
| `composables/useAgentKernel.ts` ~L32 | `callAI()` 参数 |
| `utils/aiPromptBuilder.ts` ~L87-88 | `buildMessages()` 返回/局部变量 |

同样的消息数组结构在三个文件里以匿名类型重复出现，没有命名。

**建议：** 提取为共享类型：
```ts
// 可放在 utils/aiTypes.ts 或 composables/useAiChat.ts
export interface ChatMessage {
  role: string
  content: string
}
```
影响范围小，无风险，顺手就能做。

## ⚫ 暂不动

### 4. `callAI`（编排器）与 `useAiChat.send`（调试面板）HTTP 调用重复

| 文件 | 函数 | 特点 |
|---|---|---|
| `composables/useAiChat.ts` | `send()` | Vue 响应式（loading/error/response），单次调用 |
| `composables/useAgentKernel.ts` | `callAI()` | 3 次重试 + 非响应式返回，适用 Promise.all 并行 |

两个函数都调 `/api/ai/chat`，HTTP 请求体和错误处理相似但不完全相同。

**暂不动原因：** 设计目的不同。`callAI` 是为编排器并行调用设计的——不能共享 Vue reactive ref（并行时 race），需要内置重试。而 `useAiChat.send` 是供调试面板用的薄封装，带 loading/error 状态绑定 UI。强行合并会让 `useAiChat` 变复杂且破坏并行安全特性。保持现状是合理的。
