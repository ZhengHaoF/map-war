### 4.2 eslint —— 配置缺陷，无法运行 ❌

```bash
$ pnpm --filter map-war-web run lint:eslint
... 15 个 .vue 文件全部报 "Parsing error: Unexpected token ?" / "The keyword 'interface' is reserved"
```

**根因**：`eslint.config.js` 仅启用 `pluginVue.configs['flat/essential']`，该预设只给 `.vue` 挂 `vue-eslint-parser` 拆分模板/脚本，**但没有为 `<script setup lang="ts">` 指定 TypeScript 解析器**（`@typescript-eslint/parser`）。于是 `.vue` 里的 TS 代码被默认 espree 当 JS 解析 → 遇 `interface`/`?` 即报语法错。这是**配置缺陷，不是代码缺陷**——oxlint 与 vue-tsc 都已证明这些文件语法合法。

**建议修复**（任选其一）：
1. 升级为 TypeScript 友好的预设并补解析器：
   ```js
   import tseslint from 'typescript-eslint'
   // ...
   ...tseslint.configs.recommended,
   {
     files: ['**/*.vue'],
     languageOptions: { parserOptions: { parser: tseslint.parser } },
   },
   ```
   需安装 `typescript-eslint` 并在 `files-to-lint` 块加入 `.ts`。
2. 或暂把 `lint` 脚本门禁收敛为 `oxlint` + `vue-tsc`（二者已全绿），`eslint` 作为可选格式化用。

## 五、单元测试（vitest）—— 失败 ❌

```bash
$ pnpm --filter map-war-web run test:unit
FAIL  src/__tests__/App.spec.ts > App > renders properly
Error: [🍍]: "getActivePinia()" was called but there was no active Pinia.
  at useStore .../pinia/dist/pinia.mjs
  at setup src/App.vue:10:19   (const gameStore = useGameStore())
```

**根因**：`src/__tests__/App.spec.ts` 直接 `mount(App)`，但 `App.vue` 的 `<script setup>` 顶层调用了 `useGameStore()` / `useSaveGame()`，必须存在 active Pinia。`vitest.config.ts` 没有 `setupFiles`，测试也未 `setActivePinia` / `app.use(pinia)`，故挂载即抛。

**建议修复**（最小改动）：
```ts
// src/__tests__/App.spec.ts
import { createPinia, setActivePinia } from 'pinia'
import { config } from '@vue/test-utils'
beforeEach(() => setActivePinia(createPinia()))
// 或：mount(App, { global: { plugins: [createPinia()] } })
```
> 注：测试还依赖 `canvas`（`HTMLCanvasElement.getContext` 未实现）、`RouterView` 未注册等 jsdom 限制，但**首要失败是 Pinia**，先修这个才能暴露后续。建议同时给 `vitest.config.ts` 加 `setupFiles` 统一装 Pinia。

当前全量测试为 **1 failed (1)**，CI 门禁若串接 `test:unit` 会红。

## 七、设计/健壮性复核（人工审查）

### 7.1 状态机 `stores/game.ts` —— 健壮 ✅
- **单一写者**：所有世界态变更经 `applyEvent(GameEvent)`，确定性 reducer；`initWorld()` 为唯一播种例外（注释明确）。
- **事件溯源存档**：`save()` 序列化 `eventLog`，`load()` 用 `initWorld()` 重置基线后逐事件 `applyEvent` 重放，回放期间 `isReplaying=true` 抑制重复日志/自动存档/地图 watcher。
- **快照/撤销**：`getSnapshot()`（AI 只读上下文，普通对象拷贝、不建响应式依赖）、`snapshotForUndo()/restoreUndo()`（调试单步回滚）都正确处理 `shallowRef` / Vue Proxy 的克隆（`.map` 展开而非 `structuredClone` Proxy）。
- **派生聚合**：`ownership` / `myCities` / `factionTroops` / `factionMorale` 均为 `computed`，不冗余存储。

⚠️ **脆弱点（非崩溃，建议加防御）**：`load()` 成功后置 `isReplaying=true` 并**依赖调用方随后触发 `requestMapReload()`（++reloadToken）来复位 `isReplaying`**。`useSaveGame.loadGame()` 遵守了此契约（load 后调 `requestMapReload`），但任何「直接调 `store.load()` 而忘记 reload」的代码路径都会让 `isReplaying` 永久卡 `true`（抑制归属 watcher 自动重绘、抑制 `dateAdvance` 自动存档）。建议把 `isReplaying` 复位收进 `load()` 自身（收尾逻辑若必须在组件侧，至少加 `console.warn` 防御提示）。

### 7.2 指令层 `utils/gameOrders.ts` —— 边界清晰 ✅
- `executeOrder(order)` 是唯一指令入口，内部按 `OrderType` 分发；PixiJS 句柄（战斗 `BattleHandle`）留模块本地，不进 store（与 `game.ts` 注释一致）。
- `applyEvent` 经各写回函数落地，事件溯源成立。

### 7.3 地图 `components/LeafletMap.vue` —— 工程量大、结构清晰 ⚠️
- 相机（橡皮筋缩放/补间/ECS 中断）、图层切换、点击命中测试、标签层、右键菜单、面板联动（focusTarget watch）实现完整，含无障碍降级（`prefers-reduced-motion`）。
- 原 4 处类型错误（drawFeature 坐标层级、多余 `mapScale` 参数、索引兜底、`unknown` 推导）**已确认修复**（见 §八）。
- 体量偏大（~1300 行），`onMounted` 内注册了 8+ 全局 `window` 事件，`onUnmounted` 已对称移除；但 `requestAnimationFrame`（`zoomRestRaf`/`cameraRaf`）在组件卸载时**未显式 `cancelAnimationFrame`**——快速 HMR/路由切换理论上存在一帧回调打到已销毁 `app` 的风险（`loadLayer` 已有 `if (!app?.renderer) return` 防御，实际触发概率低）。

### 7.4 AI 编排层（前端）`composables/*` + `utils/ai*` —— 职责清晰 ✅
- `useAiChat`（薄 HTTP 封装）→ `useAiDebug`（send→parse→validate→execute）→ `aiOrderContract`（12 条 `ORDER_TYPES` 契约 + 三层校验：结构/玩家战略硬编码/世界 AI 可行性）→ `executeOrder`。

- 4 处 `as any` 均在 LLM 原始 `unknown` 回包解析处（`useAiDebug.ts:84/309`、`PlayerAiPanel.vue:188/194`），属「外部不可信输入」的合理收窄，可改为 `as ChatCompletion` 类型收紧但不阻塞。
