# 汇文明朝体 — 字体动态切片实施方案

> 路线 A：预处理拆分子集 + CSS `unicode-range` 按需加载

---

## 1. 现状

| 项目 | 值 |
|------|-----|
| 字体文件 | `apps/web/src/assets/fonts/汇文明朝体.otf` |
| 文件大小 | **24 MB** |
| 格式 | OTF（OpenType/CFF） |
| 使用方式 | 尚未挂载到 `@font-face`；当前 `--font-kai` 回退系统字体 |
| 预期用途 | 全局中文 UI（`--font-kai` 覆盖 15+ 组件） |

**问题**：24MB 直接打成一个 `@font-face`，首屏字体下载耗时数秒，体验不可接受。

---

## 2. 方案原理

```
                  ┌──────────────────────────────┐
                  │       汇文明朝体.otf (24MB)     │
                  └──────────────┬───────────────┘
                                 │ pyftsubset / fonttools
                  ┌──────────────┼───────────────┐
                  ▼              ▼               ▼
          ┌──────────┐   ┌──────────┐    ┌──────────┐
          │ Latin +  │   │ CJK A    │    │ CJK B    │  ...
          │ Symbols  │   │ U+4E00-  │    │ U+7000-  │
          │ .woff2   │   │ 6FFF     │    │ 9FFF     │
          └────┬─────┘   └────┬─────┘    └────┬─────┘
               │              │               │
               ▼              ▼               ▼
          @font-face {   @font-face {    @font-face {
            unicode-range: unicode-range:  unicode-range:
            U+20-7E,...   U+4E00-6FFF     U+7000-9FFF
          }              }               }
                                 │
                                 ▼
                     浏览器只下载页面实际用到
                     的 unicode-range 对应的子集
```

核心思想：**把 24MB 的大字体拆成 N 个小份，每份覆盖一段 Unicode 区间**。CSS `@font-face` 声明 `unicode-range`，浏览器自动判断当前页面用到了哪些字符，只下载覆盖那些字符的子集文件。

这是 Google Fonts、Adobe Fonts 等 CDN 分发 CJK 字体的标准做法。

---

## 3. 分片策略

### 3.1 设计原则

- **首片要小**：Latin + 标点 + 数字（几乎所有页面都需要），控制在 50KB 以内
- **CJK 按频次拆**：常用字在前，生僻字在后，页面大概率只命中前 1-2 片
- **单片不超过 3MB**：保证 4G 网络下单片 2-3 秒可下载完

### 3.2 推荐分片方案（共 6 片）

| 片号 | 覆盖范围 | 字符数 | 预计 woff2 大小 | 命中场景 |
|------|----------|--------|----------------|----------|
| **0** | Latin + 数字 + 标点 + 全角符号 | ~500 | **~40 KB** | 所有页面 |
| **1** | CJK 高频字 U+4E00–U+62FF | ~5,000 | **~2.0 MB** | 90% 中文 UI |
| **2** | CJK 中频字 U+6300–U+77FF | ~5,000 | **~2.0 MB** | 少数生僻地名 |
| **3** | CJK 低频字 U+7800–U+9FFF | ~5,000 | **~2.0 MB** | 极生僻字 |
| **4** | CJK 扩展 A U+3400–U+4DBF | ~6,500 | **~2.5 MB** | 古籍/异体 |
| **5** | 〇 + 补充符号 | 少量 | **~50 KB** | 特殊符号 |

> **说明**：U+4E00–U+9FFF 共 20,992 个码点，按 5K 一片切 4 片合理。CJK 扩展 A（U+3400–U+4DBF）在当前项目中极大概率用不上，浏览器不会下载 —— 这正是 `unicode-range` 的价值。

### 3.3 unicode-range 精确写法

```css
@font-face {
  font-family: 'HuiWen Ming';
  src: url('/fonts/huiwen-latin.woff2') format('woff2');
  unicode-range: U+0020-007E, U+00A0-00FF, U+2000-206F,
                 U+3000-303F, U+FF00-FFEF, U+2018-2019,
                 U+201C-201D, U+2013-2014, U+2026;
  font-display: swap;
}

@font-face {
  font-family: 'HuiWen Ming';
  src: url('/fonts/huiwen-cjk-1.woff2') format('woff2');
  unicode-range: U+4E00-62FF;
  font-display: swap;
}

/* ... 余下同理 */
```

关键 CSS 属性：
- **`font-display: swap`**：立即用回退字体渲染，字体加载完再替换，避免 FOIT（不可见文本闪烁）
- 所有 `@font-face` 共享同一 `font-family` 名，浏览器自动按 `unicode-range` 拼合

## 4. 工具链选型

### 4.1 对比

| 工具 | 语言 | 优点 | 缺点 |
|------|------|------|------|
| **fonttools (pyftsubset)** | Python | 行业标准、功能最全、CJK 支持最好 | 需 Python 环境 |
| fontmin | Node.js | 纯 JS、无外部依赖 | 维护弱、CJK 处理可能丢字形 |
| harfbuzz-subset | C/Rust | 速度快 | 安装复杂、生态不够成熟 |
| glyphhanger | Node.js | 自动扫描页面用到的字符 | 对 SPA 不友好 |

### 4.2 结论：**fonttools (pyftsubset)**

理由：
- 它是 Google Fonts 后端使用的同款工具
- 对 OTF→woff2 转换和 unicode-range 切片有一流支持
- 项目已有 Python 环境（`scripts/` 下有 Python 脚本）
- 导出 woff2 一步到位：`pyftsubset input.otf --unicodes=... --output-file=output.woff2 --flavor=woff2`

## 5. 实施步骤

### Step 0 — 环境准备

```bash
# 安装 fonttools（含 pyftsubset + brotli 压缩）
pip install fonttools[woff] brotli
```

- `fonttools[woff]`：提供 woff2 编译能力（依赖 `brotli`）
- `brotli`：woff2 使用的压缩算法

### Step 1 — 编写切片脚本

新建 `scripts/subset-font.py`：

```python
#!/usr/bin/env python3
"""汇文明朝体 unicode-range 切片脚本"""
import subprocess
import os

FONT_SRC  = "apps/web/src/assets/fonts/汇文明朝体.otf"
OUT_DIR   = "apps/web/public/fonts/huiwen/"

# unicode-range 分片定义
SLICES = [
    ("latin",    "U+0020-007E,U+00A0-00FF,U+2000-206F,"
                "U+3000-303F,U+FF00-FFEF,U+2018-2019,"
                "U+201C-201D,U+2013-2014,U+2026"),
    ("cjk-1",    "U+4E00-62FF"),
    ("cjk-2",    "U+6300-77FF"),
    ("cjk-3",    "U+7800-9FFF"),
    ("cjk-ext-a","U+3400-4DBF"),
    ("cjk-misc", "U+3007,U+00B7,U+FF0C,U+3001,U+3002,"
                 "U+2018-2019,U+201C-201D"),  # 〇·，。"" 等
]

os.makedirs(OUT_DIR, exist_ok=True)

for name, unicodes in SLICES:
    out_path = os.path.join(OUT_DIR, f"huiwen-{name}.woff2")
    cmd = [
        "pyftsubset", FONT_SRC,
        f"--unicodes={unicodes}",
        f"--output-file={out_path}",
        "--flavor=woff2",
        "--layout-features=''",     # 去掉连字/kerning 等无关特性
        "--no-hinting",             # 去掉 hinting 数据
        "--desubroutinize",         # 展开子程序，利于压缩
    ]
    subprocess.run(cmd, check=True)
    size_kb = os.path.getsize(out_path) / 1024
    print(f"  ✓ {name:12s}  →  {size_kb:8.1f} KB")

print("\nDone. All subsets written to", OUT_DIR)
```

### Step 2 — 运行切片

```bash
python scripts/subset-font.py
```

预期输出：
```

  ✓ cjk-1        →   1987.3 KB
  ✓ cjk-2        →   2011.5 KB
  ✓ cjk-3        →   1892.8 KB
  ✓ cjk-ext-a    →   2401.6 KB
  ✓ cjk-misc     →     15.7 KB
```

### Step 4 — 挂载到全局样式

修改 `apps/web/src/App.vue` 顶部 `<style>` 块：

```css
@import '@/assets/fonts/huiwen.css';

/* ... 然后在 :root 中更新 --font-kai */
:root {
  --font-kai: 'HuiWen Ming', 'KaiTi', 'KaiTi_GB2312', 'SimSun', serif;
}
```

或者直接在 `main.ts` 中 `import './assets/fonts/huiwen.css'`。

### Step 5 — 构建集成

在 `package.json` 中添加：

```json
{
  "scripts": {
    "subset-font": "python scripts/subset-font.py",
    "prebuild": "pnpm subset-font",
    "build": "vite build"
  }
}
```

> 字体切片产物放在 `public/fonts/` 因为文件名固定（不走 Vite hash），且需要跨构建复用（字体不常变，无需每次 build 重切）。

## 7. 性能预估

### 7.1 首屏字体加载

| 场景 | 加载文件 | 总大小 | 对比 24MB OTF |
|------|---------|--------|---------------|
| 纯英文/数字 UI | `huiwen-latin.woff2` | **~40 KB** | **600x 缩小** |
| 典型中文 UI（< 5000 个不重复汉字） | latin + cjk-1 | **~2 MB** | **12x 缩小** |
| 含生僻地名（> 5000 字） | latin + cjk-1 + cjk-2 | **~4 MB** | **6x 缩小** |
| 全字符集 | 全部 6 片 | ~8.4 MB | 3x 缩小（且按需分散加载） |

### 7.2 浏览器行为

- **`font-display: swap`**：页面先用 KaiTi 回退字体立即渲染，woff2 下载完成后自动替换
- **并行下载**：不同 unicode-range 的子集文件由浏览器并行 fetch
- **缓存**：子集文件名固定，CDN/浏览器可长期缓存
- **按需命中**：CJK 扩展 A（2.5MB）在正常 UI 中不会被下载

## 8. 注意事项

### 8.1 `font-display: swap` 的副作用

页面首次渲染会短暂显示回退字体（KaiTi），字体加载完后替换。如果 KaiTi 和 HuiWen Ming 的度量（metrics）差异大，可能导致**布局抖动**（CLS）。

**缓解手段**：通过 `size-adjust` 微调回退字体的尺寸比例，使两字体占位接近：

```css
@font-face {
  font-family: 'HuiWen Ming fallback';
  src: local('KaiTi');
  size-adjust: 98%;  /* 根据实测微调 */
}
```

此项可在上线后根据实际效果决定是否做。

### 8.2 子集产物是否进 Git

| 策略 | 推荐度 | 说明 |
|------|--------|------|
| **进 Git + LFS** | ⭐⭐⭐ | 字体不常变，track 产物避免 CI 装 Python；8MB 总大小可接受 |
| **`.gitignore` + CI 生成** | ⭐⭐ | 干净，但 CI 需要 Python 环境 |
| **进 Git（不进 LFS）** | ⭐ | 简单但仓库会膨胀 |

推荐 **进 Git（不进 LFS）**，总大小 8MB 对一个前端仓库完全可接受。

### 8.3 woff2 兼容性

woff2 被所有现代浏览器支持（Chrome 36+, Firefox 39+, Safari 12+, Edge 14+），覆盖率达 **97%+**。项目已不需要支持 IE，无需加 woff 回退。

### 8.4 Pyftsubset 可能丢失的内容

`--layout-features='' --no-hinting --desubroutinize` 三个 flag：
- 去掉了 OpenType 排版特性（连字、kerning、竖排等）— 游戏 UI 不需要
- 去掉了 hinting 数据 — 现代渲染引擎（DirectWrite/CoreText）基本不用
- 展开了 CFF subroutines — 略微增大 woff2 但提高解压速度

如果后续发现某些字形显示异常（极少见），去掉 `--layout-features=''` 即可。

### 8.5 字体授权

汇文明朝体的授权协议需要确认是否允许**格式转换（OTF→woff2）**和**子集化**。大部分开源中文字体（如思源系列、霞鹜系列）允许此操作，但如果是商业字体需要检查授权条款。

## 9. 后续优化（可选）

| 方案 | 说明 | 收益 |
|------|------|------|
| **Glyphhanger 精筛** | 先扫描所有 `.vue/.ts` 文件统计实际用到的汉字，只切那些字符 | cjk-1 可能从 2MB 降至 1MB |
| **可变字体 (VF)** | 如果字体有 VF 版本（汇文大概率没有），用 woff2 VF | 一份文件覆盖所有字重 |
| **Service Worker 缓存** | SW 拦截字体请求做更激进缓存 | 二次访问 0 网络请求 |
| **CDN 推送** | `<link rel="preload">` 预加载 latin 子集 | 首屏无等待 |

## 10. 总结

| 维度 | 结论 |
|------|------|
| **实施复杂度** | 中低（一个 Python 脚本 + 一个 CSS 文件） |
| **首屏字体体积** | 24MB → **~2MB**（典型场景 12x 优化） |
| **浏览器兼容性** | ✅ 97%+ 支持 |
| **对现有代码侵入** | 极小（只改 `--font-kai` 变量值） |
| **回退兼容** | 字体加载失败自动回退 KaiTi，不影响可读性 |
